# Firebase-First Data Architecture Design
## Sapthala Boutique Platform

### Executive Summary
This document outlines the migration from MongoDB-centric architecture to **Firebase-first** (Firestore + Firebase Auth + Cloud Functions), making the platform real-time, more scalable, and easier to deploy globally.

---

## 1. Current State (MongoDB-Based)
**Authentication**: Firebase Auth (client-side only)
**Database**: MongoDB (Mongoose schemas)
**Backend**: Node/Express with manual Firebase token verification
**Real-time**: None (polling-based)

### Current Collections:
- `users` (accounts with roles)
- `staff` (workforce management)
- `orders` (order tracking)
- `customers` (customer profiles)
- `branches` (multi-branch support)
- `vendors` (vendor management)
- `settings` (app configuration)
- `notifications` (notifications)
- `loginAttempts` (audit logs)

**Firestore Security Rule**:
```
match /users/{userId} {
  allow read: if request.auth.uid == userId || 
                get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'super-admin';
  allow write: if request.auth.uid == userId || 
                 get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'super-admin';
  allow create: if get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'super-admin';
```

## 4. Cloud Functions (Node.js)

### 4.1 Auth Triggers
**Function**: `initializeUserDocument`
- **Trigger**: `firebase.auth().onCreate()`
- **Action**: Create user document in Firestore with default permissions
- **Code**:
```javascript
exports.initializeUserDocument = functions.auth.user().onCreate((user) => {
  const db = admin.firestore();
  return db.collection('users').doc(user.uid).set({
    email: user.email,
    phoneNumber: user.phoneNumber || null,
    role: 'customer', // default role
    permissions: {},
    isActive: true,
    createdAt: admin.firestore.FieldValue.serverTimestamp(),
    lastLogin: null
  });
});
```

### 4.2 Order Triggers
**Function**: `assignOrderTasks`
- **Trigger**: `firestore.document('orders/{orderId}').onWrite()`
- **Action**: Auto-create workflow tasks, assign staff
- **Impact**: Real-time task assignment without API call

**Function**: `updateOrderStatus`
- **Trigger**: `firestore.document('orders/{orderId}/timeline/{eventId}').onCreate()`
- **Action**: Update parent order's currentStage based on last task completion

### 4.3 Notification Triggers
**Function**: `sendNotificationToStaff`
- **Trigger**: `firestore.document('notifications/{notificationId}').onCreate()`
- **Action**: Send FCM push notification; update staff's notification badge
- **Real-time**: Uses Cloud Messaging (replaces polling)

## 5. Security Rules

### 5.1 Firestore Rules
```rules
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Super-admin: full access
    match /{document=**} {
      allow read, write: if isAdmin();
    }

    // Users: own profile + other constraints
    match /users/{userId} {
      allow read: if isOwnerOrAdmin(userId);
      allow write: if isOwnerOrAdmin(userId) && validateUserUpdate();
    }

    // Orders: admins + staff assigned to it
    match /orders/{orderId} {
      allow read: if isAdmin() || isStaffAssignedToOrder(orderId);
      allow write: if isAdmin();
      
      // Timeline: staff can append
      match /timeline/{eventId} {
        allow create: if isAuthenticated();
        allow read: if isAdmin() || isStaffAssignedToOrder(orderId);
      }
    }

    // Staff: visible to admins + their own profile
    match /staff/{staffId} {
      allow read: if isAdmin() || ownStaffProfile(staffId);
      allow write: if isAdmin();
    }
  }

  // Helper functions
  function isAuthenticated() {
    return request.auth != null;
  }

  function isAdmin() {
    return get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role in ['super-admin', 'admin'];
  }

  function isOwnerOrAdmin(userId) {
    return request.auth.uid == userId || isAdmin();
  }

  function validateUserUpdate() {
    let incomingData = request.resource.data;
    let existingData = resource.data;
    // Super-admin can change anything; others can only update certain fields
    return isAdmin() || 
           (incomingData.keys().hasAll(['email', 'name']) && 
            !incomingData.keys().hasAny(['role', 'permissions']));
  }

  function isStaffAssignedToOrder(orderId) {
    let staffId = getUsersStaffId(request.auth.uid);
    return staffId != null && 
           get(/databases/$(database)/documents/orders/$(orderId)).data.workflowTasks.some(t => t.assignedToStaffId == staffId);
  }

  function getUsersStaffId(uid) {
    // Staff members have their UID linked in staff collection
    let staffDoc = firestore.collection('staff').where('firebaseUid', '==', uid).limit(1);
    return staffDoc.length > 0 ? staffDoc[0].staffId : null;
  }

  function ownStaffProfile(staffId) {
    // If staff member is logged in via Firebase, verify they own this profile
    // This requires staff to have a firebaseUid field
    return get(/databases/$(database)/documents/staff/$(staffId)).data.firebaseUid == request.auth.uid;
  }
}
```

## 6. Migration Strategy (Phase-wise)

### Phase 1: Setup & Validation (Week 1)
1. **Create Firestore database** in Firebase console
2. **Enable Cloud Storage** for image uploads
3. **Deploy Firestore Rules** (security layer)
4. **Create Cloud Functions project** (gcp/functions)
5. **Validate**: All collections exist, security rules applied

### Phase 2: Data Migration (Week 2)
1. **Create migration utility** (Node.js script)
   - Read from MongoDB
   - Transform to Firestore schema
   - Batch write to Firestore (500 docs per batch)
2. **Migrate core collections** (in order):
   - `users` (map MongoDB `_id` to Firestore `uid`)
   - `branches`
   - `staff` (ensure staffId remains primary key)
   - `vendors`
   - `settings`
3. **Migrate transactional data**:
   - `orders` (with nested `workflowTasks`)
   - `customers`
4. **Verify**: Consistency checks (count, sample docs)

### Phase 3: Backend Update (Week 3)
1. **Create Firestore DAL** (Data Access Layer)
   - Replace Mongoose with Firestore SDK
   - Implement query builders
2. **Update API endpoints**:
   - Switch from MongoDB queries to Firestore
   - Maintain existing response format for frontend compatibility
3. **Implement Firestore indexing**:
   - Create composite indexes for multi-field queries (branch + status, etc.)

### Phase 4: Cloud Functions Deployment (Week 4)
1. **Deploy auth triggers** (user initialization)
2. **Deploy order triggers** (task auto-creation)
3. **Deploy notification triggers** (FCM integration)
4. **Test**: E2E workflows with functions

### Phase 5: Frontend Real-Time Updates (Week 5)
1. **Implement Firestore listeners** in React/Flutter
2. **Real-time dashboard** (orders update as tasks progress)
3. **Real-time notifications** (via FCM)

## 8. Benefits of Firebase-First

| Aspect | MongoDB | Firestore |
|--------|---------|-----------|
| **Scalability** | Manual sharding | Auto-scaling ✅ |
| **Real-time** | Polling required | Built-in listeners ✅ |
| **Auth Integration** | Manual JWT mapping | Native Firebase ✅ |
| **Cost at scale** | Increasing | Per-read/write ✅ |
| **Global deployments** | CDN setup needed | Multi-region ✅ |
| **Offline support** | Not built-in | SDK support ✅ |
| **Security Rules** | App-level enforcement | Enforced by DB ✅ |
| **Backup/Recovery** | Manual setup | Automatic ✅ |

---

## 9. Rollback Plan

If Firestore migration causes issues:
1. **Keep MongoDB running** in parallel (3-month window)
2. **Dual-write mode**: Write to both MongoDB and Firestore simultaneously
3. **Switch back**: If issues, revert backend to MongoDB without data loss
4. **Gradual cutover**: Move customer cohorts to Firestore incrementally
